# E3 — Manifest fetch returns HTML

## Symptoms

- "`Manifest: Line: 1, column: 1, Syntax error`."
- "PWA install prompt is missing."
- "`manifest.json` parse error in console."

## Failing layer

`<link rel="manifest" href="/manifest.webmanifest">` resolves to a URL
that responds with `text/html` instead of JSON. Common causes:

- Unauthenticated `/manifest.webmanifest` redirects to a login HTML
  page.
- SPA fallback rewrite catches `*.webmanifest` and serves
  `index.html`.
- The file simply isn't deployed; the server returns its 404 page (as
  HTML).

## Probe sequence

### 1. Find the console error

```bash
ff-rdp console \
  --filter 'Manifest|webmanifest' \
  --limit 5 --jq '.results'
```

**signal → conclude (preliminary):** syntax error mentioning the
manifest URL → suspect manifest URL identified.

### 2. Inspect the response

Preferred:

```bash
ff-rdp network --filter 'manifest|webmanifest' --detail --headers --limit 3
```

Fallback:

```bash
ff-rdp network --filter 'manifest|webmanifest' --detail --limit 3
curl -i '<manifest-url>'
```

**signal → conclude:** response `Content-Type: text/html` (or 302 →
HTML) → **manifest URL returns HTML**. Diagnosis confirmed.

**narrow → next step:** response is `application/manifest+json` but
malformed → server-side manifest content bug; out of scope of this
playbook.

### 3. Walk the redirect chain

```bash
ff-rdp network --filter 'manifest' --detail --limit 10 \
  --jq '.results[] | {url, status, content_type: .response_headers["Content-Type"]}'
```

**signal → conclude:** chain ends at `/login` → auth-gated manifest;
the manifest must be reachable unauthenticated.

**signal → conclude:** chain ends at `/index.html` → SPA fallback is
catching the request; configure the server to exempt
`*.webmanifest` (and `/manifest.json`, `/sw.js`, etc.) from the SPA
fallback rule.

## Red herrings

- `Manifest: Property '...' is required` errors → the manifest *is*
  JSON, but missing fields. Not this playbook.
- `applicationCache` deprecation warnings — unrelated.

## Worked example

Fixture: `evals/fixtures/E3/`. The fixture's `manifest.webmanifest`
file contains `<html>...</html>` content (intentionally broken).
Firefox logs a manifest parse error on load; the network response's
`Content-Type: text/html` matches the diagnosis.

## Fallback if iter-57 flags missing

`--headers` is preferred but `curl -i` is a clean alternative.
