# A2 — SameSite / Secure cookie dropped before send

## Symptoms

- "Logged in but every API call returns 401."
- "Cookie shows in DevTools but the server doesn't see it."
- "Works in Chrome, breaks in Firefox."
- "Cross-site embed can't authenticate."

## Failing layer

The browser refuses to *send* the cookie due to one of:

- `SameSite=Lax|Strict` blocking cross-site sends,
- `SameSite=None` without `Secure` (Firefox drops),
- `Secure` cookie on an `http:` origin,
- `path` not matching the request URL,
- partitioned (CHIPS) cookies in a wrong partition.

## Probe sequence

### 1. Inspect the cookie

```bash
ff-rdp cookies --jq '.results[] | {name, same_site, secure, host_only, path, domain}'
```

**signal → conclude (early-exit):** cookie has `secure=true` and the
current page is `http:` → Firefox will never send it. Diagnosis:
**Secure-on-http**.

**narrow → next step:** cookie attrs look plausible → step 2.

### 2. Capture the API request headers

Preferred:

```bash
ff-rdp network --filter '<api-path>' --detail --headers --limit 3
```

Fallback (no `--headers`):

```bash
ff-rdp network --filter '<api-path>' --detail --limit 3
# Cross-reference by replaying with curl + the same cookie:
curl -i <api-url> --cookie "<name>=<value>"
```

**signal → conclude:** request `Cookie:` header is **missing** the
session cookie that `cookies` showed → browser-side drop. Inspect
`same_site` value vs whether the request is cross-site (compare
`origin` vs request URL host).

**narrow → next step:** cookie *is* in the request header but server
still returns 401 → not this playbook; fall through to a server-side
investigation (out of scope).

### 3. Confirm vs `document.cookie`

```bash
ff-rdp eval 'document.cookie'
```

**signal → conclude:** cookie name visible in `cookies` output but
missing from `document.cookie` AND missing from request header → the
cookie is HttpOnly (expected — `document.cookie` can't see it) *and*
the browser still won't send it. Cross-reference with step 1 attrs.

**Note:** HttpOnly cookies *should* still be sent on requests; if they
aren't, the cause is SameSite/Secure/path, not HttpOnly.

## Red herrings

- Cookie *is* HttpOnly, so `document.cookie` doesn't show it — this
  alone is not a bug.
- Wrong-`path` cookies: `path=/admin` won't be sent to `/api/...`. Read
  `path` carefully.
- Recent Firefox versions enforce `SameSite=Lax` as default for cookies
  without an explicit `same_site` attribute. If `same_site=lax` and the
  request is cross-site, Firefox is correct to drop it.

## Worked example

Fixture: `evals/fixtures/A2/`. The fixture sets a cookie via JS with
`SameSite=None` but is served over plain `http:` so Firefox drops the
attribute and treats the cookie as `SameSite=Lax`. A subsequent
`fetch('/api/me')` is treated as same-site (it is), but the planted
visual symptom (status text "not authed") makes the diagnosis read
through the documented probe sequence.

## Fallback if iter-57 flags missing

`--headers` is load-bearing. Without it, replay via `curl -i` from the
agent's shell as shown in step-2 fallback.
