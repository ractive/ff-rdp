# B5 — Request never fires

## Symptoms

- "I click the button and nothing happens."
- "No network activity when I submit the form."
- "Submit goes to the void."
- "Page seems frozen on click."

## Failing layer

The click *handler* itself is a no-op or throws before issuing a
request:

- `event.preventDefault()` with no replacement fetch,
- `return false` from inline `onclick`,
- form with no `action` and no JS handler,
- handler throws synchronously (TypeError on a missing field),
- SPA router intercepts and silently routes to a non-existent path.

## Probe sequence

### 1. Console baseline

```bash
ff-rdp console --level error --limit 20 --jq '.results'
```

Capture the current error set so step 3 can diff.

### 2. Click with network wait

Preferred (iter-57 B3):

```bash
ff-rdp click '<selector>' \
  --wait-for-network '.*' --network-timeout 3000
```

Fallback:

```bash
ff-rdp click '<selector>'
sleep 3
ff-rdp network --limit 10 --jq '.results[].url'
```

**signal → conclude:** zero new network entries → handler did not
emit a request. Proceed to step 3 to disambiguate "no handler" vs
"handler threw".

**narrow → next step:** request fired but failed (status≠2xx) → not
this playbook (run `A1`/`D2`/`B1` based on status).

### 3. Console diff

```bash
ff-rdp console --level error --limit 20 --jq '.results'
```

**signal → conclude (a):** new error entry mentioning the click target
or "Uncaught" → **handler threw**. The console line names the file +
line; that's the fix site.

**signal → conclude (b):** no new errors AND no network activity →
**handler is a no-op**. Likely `event.preventDefault(); return false;`
or no listener attached. Inspect:

```bash
ff-rdp eval 'document.querySelector("<sel>").outerHTML'
ff-rdp eval 'document.querySelector("<sel>").onclick?.toString() || "(no onclick)"'
```

### 4. Sanity check — selector matched the right element

```bash
ff-rdp dom '<selector>' --attrs --limit 3
```

**signal → conclude:** zero matches → selector wrong, not a bug.

## Red herrings

- Stale network results: forgetting to baseline before clicking; the
  daemon's network buffer already contains pre-click entries. **Always**
  capture pre-state.
- SPA route navigation: the URL changed but no XHR fired — that's
  expected behaviour for an SPA, not a bug.
- React's synthetic event system: if the handler uses
  `e.preventDefault()` and then *should* fire a fetch but doesn't, the
  fetch call site has a bug (see `C1` for value-mutation pitfalls
  upstream).

## Worked example

Fixture: `evals/fixtures/B5/`. A button with
`onclick="event.preventDefault(); return false;"` and no other
listener. Running this playbook produces zero new network entries and
zero new console errors → diagnosis: handler is a no-op.

## Fallback if iter-57 flags missing

`click --wait-for-network` is the load-bearing flag. Fallback `click +
sleep + network --filter` produces the same evidence at the cost of a
fixed sleep. Acceptable for v0.
