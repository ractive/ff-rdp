# C1 — React onChange not fired by direct value mutation

## Symptoms

- "I set the input but the form thinks it's empty."
- "`ff-rdp type` fills the field but the dropdown / autocomplete
  doesn't appear."
- "Value shows in the DOM but state is stale."
- "Submit button stays disabled even though the field has text."

## Failing layer

React tracks input values via a getter/setter installed on the native
`HTMLInputElement.prototype`. Setting `input.value = "x"` directly
mutates the DOM but **bypasses** React's setter, so React's internal
`_valueTracker` doesn't see the change and `onChange` never fires. The
same applies to Preact, Solid in compat mode, and any framework using
the native value-setter pattern.

## Probe sequence

### 1. Type into the field

```bash
ff-rdp type '<selector>' 'test-value'
```

### 2. Confirm the DOM value is set

```bash
ff-rdp eval 'document.querySelector("<sel>").value'
```

**narrow → next step:** value matches input → DOM mutation worked;
step 3.

**signal → conclude:** value is empty / mismatched → not this
playbook (selector is wrong or input is read-only).

### 3. Verify the framework reacted

Pick the most relevant of these to your app:

```bash
# Autocomplete: is the suggestions popover open?
ff-rdp dom '[role=listbox], .autocomplete-suggestions' --limit 3
# Submit button: still disabled?
ff-rdp dom '<submit-sel>' --attrs --jq '.results[] | .attrs.disabled'
# Generic: does the value survive a re-render? trigger one:
ff-rdp eval 'document.querySelector("<sel>").value'
# (read again after a tick)
```

**signal → conclude:** DOM `.value` correct **but** the
framework-dependent indicator (open listbox, enabled button, mirrored
state) hasn't changed → **React doesn't see the event**. Diagnosis is
C1.

**narrow → next step:** indicator did change → not this playbook.

### 4. Confirm by dispatching the synthetic event manually

```bash
ff-rdp eval '
  (() => {
    const el = document.querySelector("<sel>");
    const setter = Object.getOwnPropertyDescriptor(window.HTMLInputElement.prototype, "value").set;
    setter.call(el, "test-value");
    el.dispatchEvent(new Event("input", { bubbles: true }));
    return el.value;
  })()
'
```

If after this the dependent indicator updates, you've confirmed C1
definitively. Recommended fix is for the agent's tooling: use a
`type` variant that dispatches via the native setter, or call the
above eval as a workaround.

## Red herrings

- "Maybe React just hasn't rerendered yet" — React renders on the next
  microtask after a real event; if it doesn't render at all, the event
  was never observed.
- Confusing `change` and `input` events — React listens to `input` for
  text inputs, `change` for checkboxes/radios.
- An input that's actually `readOnly` or wrapped in `<fieldset
  disabled>` — those silently swallow writes; step 2 detects this.

## Worked example

Fixture: `evals/fixtures/C1/`. A vanilla-JS imitation of the React
pattern: a status `<span>` flips only on a real `input` event listener.
After `ff-rdp type`, the input's `.value` is set but the status span
still reads "empty" — diagnosis matches.
