# C3 — Consent banner intercepting clicks

## Symptoms

- "Clicks do nothing."
- "Page seems unresponsive."
- "`ff-rdp click` reports success but nothing changes."
- "There's a banner I can't dismiss."

## Failing layer

A fixed-position consent overlay (OneTrust, Didomi, Cookiebot, generic
`cmp_*`) covers the target element. Pointer events go to the overlay,
not the underlying button.

## Probe sequence

### 1. What's actually at the click point?

```bash
ff-rdp eval '
  (() => {
    const el = document.elementFromPoint(
      window.innerWidth / 2,
      window.innerHeight / 2
    );
    return { tag: el?.tagName, id: el?.id, cls: el?.className };
  })()
'
```

**signal → conclude:** returned tag is a `<div>` whose id/class
contains `onetrust`, `didomi`, `cookiebot`, `cmp`, `consent`, `truste`,
or `gdpr` → **consent overlay**. Diagnosis confirmed.

**narrow → next step:** returned element is the expected button → not
this playbook; try `B5` (handler is a no-op).

### 2. Inspect the overlay

```bash
ff-rdp dom '[id*="onetrust"], [id*="didomi"], [class*="cmp"], [class*="cookie-banner"], [class*="consent"]' \
  --attrs --limit 3
```

Lists candidates so the user can confirm which library to dismiss.

### 3. Dismiss via library hook (preferred)

```bash
# OneTrust
ff-rdp eval 'window.OneTrust?.AllowAll?.()'
# Didomi
ff-rdp eval 'window.Didomi?.setUserAgreeToAll?.()'
# Generic comparis-style flag
ff-rdp eval 'window.cmp_noscreen = true'
```

Or use `ff-rdp launch --auto-consent` next time to skip the dialog
entirely.

### 4. Confirm dismissal

```bash
ff-rdp eval '
  document.elementFromPoint(window.innerWidth/2, window.innerHeight/2).tagName
'
```

**signal → conclude:** now returns the expected interactive element →
overlay is gone; retry the failing click. If the click still does
nothing, the diagnosis was wrong; run `B5`.

## Red herrings

- A `<div>` with low opacity that's *not* a consent banner — common
  during page transitions. Cross-check by waiting 1s and re-running
  step 1; transition overlays clear themselves.
- Modal dialogs (`<dialog>`, Radix Dialog) which are real intentional
  modals, not bugs — recognise the `role=dialog` aria attribute.
- Server-side rendered banners that re-appear after dismissal because
  storage write was blocked by Tracking Protection.

## Worked example

Fixture: `evals/fixtures/C3/`. A full-screen
`<div id="onetrust-banner" style="position:fixed;inset:0;z-index:9999">`
covers a target button. Step 1 reports the banner div; step 3 dismisses
it (the fixture wires the OneTrust shim); step 4 confirms.
