# E1 — ChunkLoadError after deploy

## Symptoms

- "`ChunkLoadError`."
- "`Loading chunk N failed`."
- "Page is broken after the deploy."
- "Blank page; works after hard refresh."

## Failing layer

The browser is running an `index.html` (or `_document.html`) cached
from before the deploy. That HTML references chunks by hashed filename
(`main.abc123.js`). After the deploy those filenames no longer exist —
the new build has `main.def456.js`. Dynamic `import()` calls inside the
old runtime try to fetch the old chunk path, get 404, throw
`ChunkLoadError`.

This is **not** a service-worker bug in the common case; it's plain
HTTP caching on `index.html`.

## Probe sequence

### 1. Confirm the error

```bash
ff-rdp console --level error \
  --filter 'ChunkLoadError|Loading chunk|chunk failed' \
  --limit 5 --jq '.results'
```

**signal → conclude (preliminary):** error matches → suspect chunk
URLs are in the message. Step 2.

### 2. Find the 404'd chunk(s)

```bash
ff-rdp network --filter '/static/|/chunks/|/_next/|/assets/' \
  --detail --limit 30 \
  --jq '.results[] | select(.status == 404) | {url, status}'
```

**signal → conclude:** ≥1 hashed asset URL returns 404 → **stale
index.html referencing dead chunks**. Diagnosis confirmed.

**narrow → next step:** all chunks 200 → not E1; possibly a runtime
error in the chunk loader itself.

### 3. Rule out service-worker involvement

```bash
ff-rdp eval 'navigator.serviceWorker?.getRegistrations?.().then(r => r.map(x => x.scope))'
```

**signal → conclude:** no registrations → diagnosis is plain HTTP
cache; fix is server-side `Cache-Control: no-cache` on `index.html`.

**narrow → next step:** SW registered → run `E2` (SW serving stale
shell) before concluding.

### 4. Confirm the stale doc

```bash
ff-rdp network --filter '/$|index.html|/$' --detail --headers --limit 5
```

Look at the `Cache-Control` / `ETag` / `Age` on the initial document.
`max-age=3600` or large `Age:` confirms stale doc.

## Red herrings

- Pinning the bug on the service worker: SW is *sometimes* the cause,
  but plain HTTP cache on `index.html` is more common.
- An ad blocker returning 404 on specific chunks (rare; usually only
  hits analytics).
- A CDN purging the chunk but caching the HTML — same root cause,
  different angle: still a deploy-skew.

## Worked example

Fixture: `evals/fixtures/E1/`. `index.html` references
`/static/chunks/main.abc123.js` which doesn't exist (404). The
fixture's inline script logs a `ChunkLoadError`-shaped message on load,
and the network panel shows the 404.

## Fallback if iter-57 flags missing

`--headers` is nice-to-have for step 4 but not load-bearing; the
status-404 evidence from step 2 is conclusive on its own.
