# C2 — Custom (role-based) dropdown can't be clicked

## Symptoms

- "My selector doesn't match the dropdown option."
- "Click finds nothing."
- "Downshift / HeadlessUI / Radix combobox is unclickable."
- "Option list isn't in the DOM."

## Failing layer

The options are `<div role="option">` rendered into a portal (a
sibling of `<body>` or `<div id="portal">`), not children of the
combobox. They appear only after the combobox is opened, and have no
stable CSS class — only `role=option` and `aria-label` / text.

## Probe sequence

### 1. Look for any rendered options

```bash
ff-rdp snapshot --filter 'role=option' --limit 20
# Or:
ff-rdp dom '[role=option]' --limit 20
```

**signal → conclude:** zero matches → the options haven't been
rendered yet (portal is empty). Step 2.

**narrow → next step:** matches exist → step 3.

### 2. Open the combobox

```bash
ff-rdp click '[role=combobox], [aria-haspopup=listbox]'
ff-rdp dom '[role=option]' --limit 20
```

**signal → conclude:** options now appear → fix the agent's flow to
click the trigger first; not really a bug, a sequencing issue.

**narrow → next step:** still zero options → step 4 (focus then
keyboard).

### 3. Pick the option

```bash
ff-rdp click '[role=option][aria-label="<label>"]'
# Or, if labels aren't unique:
ff-rdp eval '
  Array.from(document.querySelectorAll("[role=option]"))
    .find(o => o.textContent.trim() === "<text>")
    ?.click()
'
```

**signal → conclude:** click succeeded and the combobox value updated
→ done.

**narrow → next step:** click "succeeded" but value didn't update →
likely C1 upstream (framework didn't see the option click event);
re-run C1 against the trigger input.

### 4. Keyboard fallback

```bash
ff-rdp click '[role=combobox]'
ff-rdp eval '
  document.activeElement.dispatchEvent(
    new KeyboardEvent("keydown", { key: "ArrowDown", bubbles: true })
  )
'
ff-rdp dom '[role=option]' --limit 20
```

**signal → conclude:** options now appear via keyboard → the trigger
opens on keydown but not click; usually accessibility config or
focus state issue.

## Red herrings

- `aria-hidden=true` portals (visually hidden but still in the
  accessibility tree) — `snapshot --filter role=option` will still
  show them; cross-check with `dom` to confirm presence.
- Options that exist but inside a `display:none` parent — they won't
  be clickable; check via `eval
  getComputedStyle(...).display`.
- Combobox using a non-standard role (`role=textbox` with custom
  `aria-controls`) — search by `aria-controls` value first.

## Worked example

Fixture: `evals/fixtures/C2/`. A combobox renders options into
`<div id="portal">` on click. Before clicking, `[role=option]` matches
nothing; after clicking the trigger, three options appear.

## Fallback if iter-57 flags missing

None — `snapshot`, `dom`, `click`, `eval` are all stable.
