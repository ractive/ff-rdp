# A1 — Set-Cookie stripped at the edge

## Symptoms

- "Login submits but I'm still logged out."
- "Auth works locally, breaks in prod."
- "Stays on the login page after a correct password."
- "Server returns 200 but no session cookie appears."

## Failing layer

CDN / reverse proxy strips the `Set-Cookie` response header on the
cached auth path, so the browser never receives the cookie even though
the auth backend issued it.

## Probe sequence

Uses the **capture-diff** primitive (`SKILL.md`).

### 1. Baseline

```bash
ff-rdp cookies --jq '.results[].name' > /tmp/pre_cookies.json
ff-rdp console --limit 20 --jq '.results' > /tmp/pre_console.json
```

### 2. Trigger login + capture network with headers

Preferred (iter-57 B2 + B3):

```bash
ff-rdp click 'button[type=submit]' \
  --wait-for-network '/api/login|/auth|/session'
ff-rdp network --filter '/api/login|/auth|/session' \
  --detail --headers --limit 5
```

Fallback (without `--wait-for-network` / `--headers`):

```bash
ff-rdp click 'button[type=submit]'
sleep 4
ff-rdp network --filter '/api/login|/auth|/session' --detail --limit 5
# Then read response headers via curl from the agent's shell:
curl -i -X POST <login-endpoint>
```

**signal → conclude:** auth request `status=2xx`, response headers
contain **no** `Set-Cookie` line → **edge stripped it**.

**narrow → next step:** auth request returned 4xx/5xx → not this
playbook (login actually failed); fall through to `A2` if 401, else
report status as the diagnosis.

### 3. Post-action capture

```bash
ff-rdp cookies --jq '.results[].name'
```

**signal → conclude:** `post_cookies == pre_cookies` (no new cookie
appeared) + step-2 response had no `Set-Cookie` → confirmed edge strip.

**narrow → next step:** new cookie appeared → cookie did arrive; if
subsequent API calls still 401, run `A2`.

### 4. Confirmation via curl

```bash
curl -i -X POST <login-endpoint> -d '<same body>'
```

**signal → conclude:** curl response also missing `Set-Cookie` → the
strip happens server- or edge-side, not in the browser. Hand off to
ops/infra to inspect the CDN config (commonly: a `Cache-Control: public`
rule on the auth path causes the CDN to drop `Set-Cookie`).

## Red herrings

- Unrelated CSP `eval` console errors (Zod v4 fires these defensively;
  the page still works).
- Intermediate 302s in the auth flow that drop the `Set-Cookie` chain —
  look at every hop, not just the final response.
- The cookie *is* set but is `HttpOnly`; the page-side
  `document.cookie` won't show it. **Always** check via `ff-rdp cookies`
  (which sees HttpOnly), not via `eval document.cookie`.

## Worked example

Fixture: `evals/fixtures/A1/`. The fixture serves `/api/login.json`
returning `{"ok":true}` with no `Set-Cookie` header. Running this
playbook against the fixture: step 2 captures the response, step 3
confirms `cookies` is empty before and after — diagnosis matches.

## Fallback if iter-57 flags missing

`--wait-for-network` and `--headers` are both load-bearing here. If
absent, see step-2 fallback block above. The diagnosis stays accurate
but takes one extra command (curl).
