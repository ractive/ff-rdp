# D2 — Trailing-slash redirect breaks fetch

## Symptoms

- "API returns HTML instead of JSON."
- "`SyntaxError: JSON.parse: unexpected character at line 1 column 1`."
- "`Unexpected token < in JSON at position 0`."
- "Worked before someone added/removed the trailing slash."

## Failing layer

A `fetch('/api/data')` is redirected (301/302/308) to a path that
returns HTML — typically because the server's trailing-slash policy
sends `/api/data` → `/api/data/` (which serves an HTML index) or
because an auth redirect bounces unauthenticated requests to a login
HTML page.

## Probe sequence

### 1. Find the parse error

```bash
ff-rdp console --level error \
  --filter 'JSON|parse|Unexpected token <|SyntaxError' \
  --limit 5 --jq '.results'
```

**signal → conclude (preliminary):** error names a fetch URL → that's
our suspect URL. Otherwise step 2 still applies.

### 2. Inspect the failing request

Preferred (iter-57 B2):

```bash
ff-rdp network --filter '<suspect-url>' --detail --headers --limit 5
```

Fallback:

```bash
ff-rdp network --filter '<suspect-url>' --detail --limit 5
curl -i '<suspect-url>'
```

**signal → conclude:** response `Content-Type: text/html` (or status
3xx with `Location:` to an HTML page) → **redirect to HTML**. Diagnosis
confirmed. The Location header names the destination.

**narrow → next step:** response is `application/json` but malformed →
not this playbook; server returned invalid JSON.

### 3. Walk the Location chain

```bash
ff-rdp network --filter '<suspect-url>' --detail --limit 20 \
  --jq '.results[] | {url, status, location: .response_headers["Location"]}'
```

**signal → conclude:** chain ends at `/login`, `/auth`, or `/` →
likely authentication redirect; cross-check `cookies` (run `A1` if the
session cookie is missing).

**signal → conclude:** chain ends at `<original>/` (added slash) → the
server enforces trailing slashes via redirect; fix is to request the
canonical URL directly.

## Red herrings

- `fetch` against an opaque CORS response — the body is unreadable but
  no parse error is thrown until `.json()` is called.
- Service workers can intercept and replace responses with HTML offline
  pages — see `E2` for that case.
- Mixed-content blocking can also produce "no body" errors that look
  like parse failures; check `console` for "Mixed Content".

## Worked example

Fixture: `evals/fixtures/D2/`. The fixture's `index.html` fetches
`/api/data` (no trailing slash); the static server serves
`/api/data.html` as HTML. `JSON.parse` then fails. The console error
"Unexpected token <" + the network response's `Content-Type: text/html`
together match the diagnosis.

## Fallback if iter-57 flags missing

`--headers` is preferred but not strictly required: even without it,
status 3xx + the redirect target on the next entry in `network` is
enough to conclude. `curl -i` is the cleanest fallback.
